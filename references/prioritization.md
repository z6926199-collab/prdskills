# 优先级评估框架

本文件整合需求优先级评估方法论，可在 Step 2 缺口澄清阶段按需加载。不影响 PRD 输出结构。

---

## 一、框架选择决策树

```
产品处于什么阶段？
├── 早期 / PMF 探索期 → ICE（轻量快速）
├── 成长期 / 有用户数据 → RICE（数据驱动）
├── 功能分类需求 → KANO 模型
└── 敏捷开发团队 → MoSCoW / Value-Effort 矩阵
```

---

## 二、KANO 模型 — 需求分类

| 需求类型 | 特征 | 策略 |
|---------|------|------|
| **基础型（必备）** | 若没有，用户极度不满 | 必须完成，作为 MVP 核心 |
| **期望型（意愿）** | 做得越好，用户越满意 | 优先完成，提升核心竞争力 |
| **兴奋型（魅力）** | 用户意想不到 | 择机完成，制造产品亮点 |
| **无差异型** | 做不做用户都没感觉 | 不做，避免浪费资源 |

**使用方式**：将功能列表按 KANO 分类后，映射到 PRD 四、需求列表的优先级（P0=基础型+期望型，P1=期望型剩余，P2=兴奋型）。

---

## 三、ICE 评分法

| 维度 | 说明 | 分值 |
|------|------|------|
| **I**mpact（影响力） | 对关键指标的影响程度 | 1-10 |
| **C**onfidence（置信度） | 我们有多确定？ | 1-10 |
| **E**ase（简易度） | 实施难度 | 1-10 |

**ICE 得分 = I × C × E**，得分越高优先级越高。

---

## 四、RICE 评分法

| 维度 | 说明 |
|------|------|
| **R**each（触达范围） | 受影响用户数 |
| **I**mpact（影响力） | 对目标指标的影响 |
| **C**onfidence（置信度） | 0-100% |
| **E**ffort（工作量） | 人月 |

**RICE 得分 = (R × I × C) / E**

---

## 五、机会评分（Dan Olsen）

```
机会评分 = 重要性 × (1 − 满意度)
```

- 归一化至 0-1 范围
- 高重要性 + 低满意度 = 最高机会
- 在「重要性 vs 满意度」图上，左上象限是黄金区域

**适用场景**：已有用户满意度数据时，用于识别最具改进空间的需求。

## 六、MoSCoW 法则

| 类别 | 含义 | 建议比例 |
|------|------|---------|
| **Must** | 必须有 | 60% |
| **Should** | 应该有 | 20% |
| **Could** | 可以有 | 15% |
| **Won't** | 本次不做 | 5% |

**映射到 PRD**：Must → P0，Should → P1，Could → P2，Won't → 四、范围中显式排除。

## 七、9大框架对比速查

| 框架 | 适用场景 | 核心特点 |
|------|---------|---------|
| 艾森豪威尔矩阵 | 个人任务 | 紧急 vs 重要 |
| 影响力 vs 工作量 | 快速分类 | 2×2 矩阵 |
| 风险 vs 回报 | 计划评估 | 考虑不确定性 |
| **机会评分** | 客户问题 | 重要性 × 满意度 |
| **KANO 模型** | 理解期望 | 需求分类 |
| 加权决策矩阵 | 多因素决策 | 标准赋权 |
| **ICE** | 快速排序 | 轻量灵活 |
| **RICE** | 规模化分析 | 增加触达范围 |
| **MoSCoW** | 需求范围 | 敏捷友好 |

## 八、使用规则

1. 用户输入含功能列表但未明确优先级时，自动激活 KANO 或 MoSCoW。
2. 用户要求「排优先级」「哪些该先做」时，根据数据可用性推荐 RICE 或 ICE。
3. 分类结果写入 PRD 四、需求列表的优先级标注（P0/P1/P2）。
4. 工具选择根据用户提供的信息丰富度自动判断，不强制显示所有框架。
