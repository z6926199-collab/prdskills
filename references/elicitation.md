# 需求澄清、场景树与证据收集

本文件覆盖起草前的全部准备工作：缺口澄清、PRD 深度确认、场景树分解、功能大纲确认、证据收集。

---

## 一、快速通道

如果用户输入已覆盖：需求背景 + 功能列表 + 目标用户 + 至少 2 项（竞品、指标、技术约束），跳过问卷，直接进入 PRD 深度确认 → 功能大纲 → 起草。

## 二、最小输入要求

至少需要：

- 需求背景：为什么要做、解决什么问题。
- 需要实现的功能：功能列表、用户场景或模块。

两者齐全即可起草，缺失项标 `待确认`。不阻塞于作者、日期、上线计划等元信息。

## 三、澄清原则

1. **问功能行为，不问项目管理元数据**。
2. **可点击选项优先**：所有选择题以可点击选项卡片呈现，每个选项含标签+一句话描述。
3. **每次最多 4 题**：工具限制单次最多呈现 4 个问题，超过时分批发送。
4. **每题必有兜底项**：最后一个选项为"其他（请补充说明）"，允许自由输入。
5. **总题数不超过 6-7 题**，分批优于一次性倾倒。
6. **非阻塞缺口写 `待确认`**，不阻断撰写。
7. **澄清后复述确认**，用户确认后再动笔。

## 四、附件资料解析

只提取事实，不推测。无法确定的标 `待确认`。

| 附件类型 | 提取并转化为 |
|-|-|
| 竞品截图 | 功能点、交互模式 → 竞品矩阵、差异化需求 |
| NPS/用户工单 | 痛点、高频问题 → 用户研究摘要 |
| 原型图/Figma | 页面骨架、流程 → 5.2 原型说明 |
| 埋点数据/看板 | 当前指标、漏斗 → 数据现状、目标基线 |
| 会议纪要 | 已确定决策 → 跳过对应问卷题 |
| 设计规范 | 可复用组件 → 交互规则参考 |
| 历史 PRD / 产品文档 | 章节结构习惯、术语、竞品列表、写作风格 → 临时应用（不生成域文件，仅当次生效） |

**历史 PRD 自动提取**：用户提供了历史 PRD 作为参考时，即使没有要求"生成域文件"，也自动提取写作风格偏好（术语、句式、5.3 组织方式、禁用词），在当次起草中临时覆盖 `writing-style.md` 通用规则。提取后在「需求理解确认」中告知用户。

## 五、缺口澄清问卷

仅问影响功能行为、用户流程、页面逻辑、数据处理、异常处理或 5.3 内容的核心决策。

结构：3-6 道选择题（可点击选项卡片）+ PRD 深度题 + 1 道开放式补充题。

### 问卷呈现方式

每道题以可点击选项卡片呈现。有推荐项时标注"（推荐）"并放在首位。最后一项始终为兜底选项，允许自由输入。示例格式：

```
问题 1：首次进入功能时的授权时机？
选项：A. 直接进入（推荐）— 进入功能页，用到权限时再弹授权
      B. 先授权再进入 — 必须完成授权才能看到功能页
      C. 先展示再授权 — 先看到引导，操作时触发授权
      D. 其他 — 请补充说明

问题 2：异常情况的拦截强度？
选项：A. 强提醒 — 弹窗提醒但允许跳过（推荐）
      B. 阻断 — 强制处理后才能继续
      C. 仅特定版本拦截 — 按版本灰度控制
      D. 其他 — 请补充说明

问题 3：PRD 深度？
选项：A. 标准版（推荐）— 常规评审用，~4000-8000 字，覆盖 P0+P1
      B. 概念版 — 方向对齐用，~2000 字
      C. 评审版 — 正式上车用，~8000-15000 字
      D. 由你判断（按需求复杂度自动推荐） — 按需求复杂度自动推荐
```

### 问卷生成规则（统一优先级制）

不分 0-to-1 / 1-to-100 两套模板。按以下优先级顺序生成问卷，从上往下取题，凑满 3-6 题即停：

| 优先级 | 问题类型 | 适用条件 | 示例 |
|-|-|-|-|
| 1 | **目标用户** | 输入中未明确 | "目标用户是谁？A 所有用户 / B 特定人群 / C 特定场景" |
| 2 | **核心场景/功能列表** | 输入中功能不完整 | "需要实现哪些功能？"（开放题） |
| 3 | **新功能 or 迭代** | 无法判断项目类型 | "这是全新功能还是已有功能的优化？" |
| 4 | **入口/交互方式** | 影响 5.3 入口字段 | "功能入口在哪？A 首页 / B 设置 / C 语音 / D 其他" |
| 5 | **异常/降级策略** | 影响 5.3 异常场景 | "异常情况如何处理？A 弹窗提醒 / B 阻断 / C 静默降级" |
| 6 | **数据/隐私** | 涉及上云或敏感数据 | "数据存储在哪？A 本地 / B 云端 / C 混合" |
| 7 | **竞品/参考** | 用户未提及 | "有无竞品参考？" |
| 8 | **升级兼容** | 1-to-1.5/1-to-100 时 | "老版本如何处理？A 不兼容 / B 降级 / C 数据迁移" |

规则：
- 已从用户输入中获取的信息 → 跳过对应优先级的题。
- 每题必须有兜底选项「其他——请补充」。
- 5.3-Driven 的细节问题（入口前置、状态机、极限并发）在场景树阶段（Step 3）再深入，不在问卷阶段问。

## 六、PRD 深度确认（必做）

```text
PRD 深度选哪种？
A. 概念版（方向对齐用，~2000 字，仅核心章节）
B. 标准版（常规评审用，~4000-8000 字，覆盖 P0+P1）
C. 评审版（正式上车用，~8000-15000 字，含全部非功能需求）
D. 由你判断（按需求复杂度自动推荐）
```

| 用户提供的信息 | 推荐 |
|-|-|
| 仅想法，无功能列表 | A |
| 有功能列表但缺细节 | B |
| 完整列表 + 上线时间 + 多端 | C |
| 模糊 | D（默认 B） |

## 七、需求理解确认

```text
我按以下理解起草：
1. 产品背景：……
2. 核心功能：……
3. 关键决策：……
4. PRD 深度：标准版
5. 暂按以下假设：……

下一步先出功能大纲，确认后展开。
```

用户说「直接写」→ 跳过复述，将假设写入 2.1 背景说明末尾的“待确认假设”，不得新增章节。

## 八、场景树分解

澄清完成后、起草前，将功能拆解为场景树。

### 基本结构

每个场景拆为：触发条件 → 用户动作 → 系统响应 → 结束状态。

### 深度要求（每个场景必须覆盖）

1. **正常路径**：至少 1 条完整成功路径。
2. **异常分支**：每个场景至少 2 条异常路径，按以下维度扫描：
   - 网络：无网 / 弱网 / 网络切换
   - 权限：未授权 / 授权后撤销
   - 并发：重复操作 / 多端同时操作
   - 中断：来电 / 退后台 / 系统杀进程
3. **终态完整性**：每条路径必须有明确终态（成功态 / 失败态 / 降级态）。

### 场景树 → 5.3 映射规则

场景树是 5.3 的骨架，每个映射关系如下：

| 场景树元素 | 对应 5.3 位置 |
|-|-|
| 正常路径 | 「具体表现 → 用户流程」 |
| 异常分支 | 「具体表现 → 功能逻辑」+ 「异常场景」列 |
| 终态（失败/降级） | 「逆向场景/退化逻辑」列 |
| 触发前置条件 | 「功能入口/前置条件」列 |
| 极端触发（高频/并发） | 「极限场景」列 |

展示场景树后等用户确认，确认后场景树直接驱动 5.3 的切分和填写——不重复分析。

## 十、证据收集（可选）

用户提供原始素材时激活。

### 端到端流程

1. **识别素材**：用户上传/粘贴原始材料时，逐个识别类型（文档/调研/数据/竞品/技术）。
2. **分配编号**：按证据编号表分配唯一编号（E01/U01/D01/C01/T01），递增不复用。
3. **结构化记录**：按证据记录格式填写每条证据（来源、日期、原始事实、置信度）。
4. **写入产物**：
   - PRD 正文章节（2.1/2.2/2.3/2.4/三/5.3）：在使用证据的段落末尾标注 `[证据编号]`。
   - 完整证据清单放在 PRD 正文之后，标记为“模板外补充材料”；不得替换或扩写原模板“附录”结构，除非用户明确要求。
5. **关联 5.3**：5.3 的「功能」字段和「具体表现」中的关键决策，若基于证据则标注 `[E01]`；若基于假设则标注 `假设（置信度：高/中/低）`。
6. **冲突处理**：多条证据矛盾时，双方保留，在 PRD 中标注 `待确认：E01 与 C02 结论冲突`。

### 证据编号

| 前缀 | 来源类型 | 示例 |
|-|-|-|
| E | 文档类（PRD、设计稿、规范、会议纪要） | E01 |
| U | 用户研究类（访谈、问卷、NPS、工单） | U01 |
| D | 数据类（埋点、看板、A/B 测试） | D01 |
| C | 竞品类（竞品分析、市场报告） | C01 |
| T | 技术类（架构图、接口文档、SDK） | T01 |

编号唯一，不复用，递增分配。域文件可定义专用前缀。

### 证据记录格式

| 字段 | 说明 |
|-|-|
| 证据编号 | 唯一标识（E/U/D/C/T + 递增序号，域文件可定义专用前缀） |
| 来源类型 | E 文档 / U 用户研究 / D 数据 / C 竞品 / T 技术 |
| 来源位置 | 文件名、链接、报告章节 |
| 日期 | 产生或获取日期 |
| 来源人/系统 | 提供者 |
| 原始事实 | 未解释的原始观察 |
| 置信度 | 高/中/低 |

### 使用规则

- 每个 P0 需求关联证据或已批准假设。
- 证据不足标 `假设（置信度：高/中/低；待验证来源：…）`。置信度判断：近期+相关+样本充足+多源一致 = 高；部分满足 = 中；轶事/过期/矛盾 = 低。
- 冲突时双方保留，记录待确认。不臆造日期、指标、法律结论。

### 证据追溯集成

以下章节需标注证据编号：2.1 背景、2.2 数据现状、2.3 调研、2.4 竞品、三 目标、5.3 关键决策。证据汇总默认作为模板外补充材料。

## 十一、推断规则

- 粗略想法 + 直接起草 → 推断骨架并标假设。
- 零散笔记 → 整理为背景/目标/需求列表/功能段落。
- 细节冲突 → 保留最新输入，冲突进 `待确认`。
- 缺指标 → 提议候选项标 `待确认`。
- 缺竞品 → 问或留 `待补充`。
- 不影响 5.3 的问题 → 不前置询问，进 `待确认事项`。
- 会议纪要有结论 → 跳过对应题，标"已确认（来源：会议纪要）"。

## 十二、Spec 深度建议

| 功能特征 | 建议深度 |
|-|-|
| P0 + 多状态 + 用户输入 | L3-L4，全子块展开 |
| P0 + 单一路径 | L3，核心子块 |
| P1 + 辅助功能 | L2 |
| P2 + 低风险 | L1 |
