# 用户研究与需求挖掘

本文件整合用户研究方法论，可在 Step 0 需求前置分析和 Step 2 缺口澄清阶段按需加载。不影响 PRD 输出结构。

---

## 一、核心分析方法

### 1.1 5W1H 分析法

| 维度 | 核心问题 | PRD 映射 |
|------|---------|---------|
| **Who** | 谁用？（多角色、权限控制） | 2.1 背景说明 / 2.3 用户调研 |
| **Where** | 什么环境下用？（移动端/PC、网络环境） | 5.3 边界场景 |
| **When** | 什么时候用？（高频/低频） | 5.3 功能入口/前置条件 |
| **Why** | 为什么要做？（解决什么痛点） | 2.1 背景说明 |
| **What** | 具体做什么？（功能本体） | 四、需求列表 / 5.3 |
| **How** | 如何实现？（前端交互、后端逻辑） | 5.3 具体表现 / 功能逻辑 |

### 1.2 Y 模型 — 深层挖掘

```
用户需求 (What) → 目标动机 (Why) → 人性本质 → 产品方案 (How)
     ↑                              ↑                   ↓
     ←──── 验证核心假设 ←───────────────────────────────
```

- 用户提出的是解决方案（Y 的上端）
- 需要深挖背后的真实动机（Y 的下端）
- 再重构更优方案

**应用**：当用户说"我想要 X 功能"时，追问"你希望通过 X 解决什么问题？"

### 1.3 JTBD 三层需求

| 需求层级 | 核心问题 | 示例 |
|---------|---------|------|
| **功能性需求** | 用户要完成什么任务？ | 自动化生成报表 |
| **社交性需求** | 用户希望别人怎么看他？ | 看起来更专业、数据驱动决策 |
| **情感性需求** | 用户想避免什么感受？ | 不再被老板质问而感到羞愧 |

## 二、伪需求识别

> "人们其实不想买一个四分之一英寸的钻头，他们只想要一个四分之一英寸的洞！"

识别伪需求的关键：

1. **追问「为什么」**：用户说"想要一匹更快的马"，真实需求是"更快的交通工具"
2. **观察用户行为**：不仅听用户说什么，更要看用户怎么做
3. **验证普遍性**：是个别用户的需求还是群体需求
4. **小范围测试**：通过 MVP 或灰度发布验证

**PRD 中的体现**：伪需求直接写入「四、不包含范围 (Out of Scope)」并注明原因。

## 三、用户访谈脚本设计

### 3.1 开场白（2-3 分钟）

- 介绍自己和访谈目的（学习，不是销售）
- 设定预期："没有对错之分，我们只是希望从您的经历中学习。"
- 征得录音许可（如适用）

### 3.2 破冰阶段（5 分钟）

- "请介绍一下您的工作职责，以及典型的一天是什么样的。"
- 目标：建立信任，理解受访者背景

### 3.3 核心探索：JTBD 访谈（15-20 分钟）

**当前情境与行为**（过去时态）：
- "请带我回顾一下您上次[事情]的经过。"
- "您用了哪些工具或方法？花了多长时间？有哪些人参与？"

**痛点与挫败感**：
- "那个过程中最难的地方是什么？"
- "如果有一根魔法棒，您希望改变什么？"

**期望产出**：
- "对您来说，这个领域'做得好'是什么样子？"

**支付意愿**：
- "您目前在这上面花了多少时间/金钱？"

### 3.4 追问技巧

- "能再说详细一点吗？" — 展开任何话题
- "为什么？" — 深挖根本原因
- "能举个具体例子吗？" — 从观点转向事实
- "接下来发生了什么？" — 顺着故事追问

### 3.5 老妈测试原则

- 询问**他们的生活**，而非你的想法
- 询问**过去的事**，而非未来
- **少说多听** — 80/20 分配
- 访谈过程中**绝不推销**
- 关注**强烈情绪** — 它们代表真实痛点

## 六、使用规则

1. 用户输入含用户反馈、访谈记录、NPS 报告、工单时激活。
2. 5W1H 分析结论直接映射到 PRD 对应章节（见 1.1 表格）。
3. 访谈结论写入 PRD 2.3 用户调研。
4. 需求分析摘要作为 Step 0.5 理解确认的输出模板之一。
5. Y 模型深挖在用户只给解决方案不给问题场景时使用。
