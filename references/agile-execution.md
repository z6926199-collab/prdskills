# 敏捷执行与数据指标

本文件整合敏捷执行层方法（OKR、Sprint、干系人、发版）和数据指标体系。在 PRD 六/七章扩展或用户要求执行规划时按需加载。不影响 PRD 输出结构。

---

## 一、OKR 与目标管理

### 1.1 OKR 模板

```
目标：让新用户享受轻松愉快的引导体验

关键成果：
- KR1：引导调研 CSAT 评分 ≥ 75%
- KR2：66% 以上的用户两天内完成引导
- KR3：平均首次价值实现时间 (TTV) ≤ 20 分钟
```

### 1.2 OKR vs KPI vs 北极星指标

| 概念 | 说明 |
|------|------|
| **OKR** | 鼓舞人心的目标 + 可衡量的关键成果 |
| **KPI** | 长期跟踪的核心量化指标 |
| **北极星指标** | 单一、以客户为中心、成功领先指标 |

**关系**：关键成果可以是 KPI；OKR 的 KR 可以表达北极星指标的变化。

### 1.3 PRD 映射

OKR 中的 KR 直接映射到 PRD 三、需求目标的指标名称和目标列。

## 二、数据指标体系

### 2.1 北极星指标框架

```
北极星指标（North Star）
├── 输入指标（3-5 个）
├── 健康指标（延迟、错误率、NPS）
└── 业务指标（MRR、CAC、LTV、流失率）
```

### 2.2 AARRR 海盗指标

| 阶段 | 指标 | 说明 |
|------|------|------|
| **A**cquisition | 获取 | 用户如何发现你？ |
| **A**ctivation | 激活 | 用户第一次体验好吗？ |
| **R**etention | 留存 | 用户会回来吗？ |
| **R**eferral | 推荐 | 用户会推荐别人吗？ |
| **R**evenue | 收入 | 你如何盈利？ |

### 2.3 好指标的 4 个标准

1. **易理解** — 形成共同语言
2. **可比较** — 随时间变化
3. **是比率** — 比绝对数字更具洞察力
4. **能改变行为** — 如果指标不会改变你的行为，它就是糟糕的指标

### 2.4 指标看板定义模板

| 指标 | 定义 | 数据来源 | 可视化 | 目标值 | 告警阈值 |
|------|------|----------|--------|--------|----------|
| [名称] | [精确计算] | [出处] | [类型] | [目标] | [条件] |

## 三、Sprint 规划

### 3.1 Sprint 规划流程

```
1. 估算团队容量（成员 × 可用时间，历史速率 3 Sprint 平均，预留 15-20% 缓冲）
2. 评审并选取故事（从优先级列表取用，验证"就绪定义"，达到容量上限停止）
3. 映射依赖关系（识别外部依赖，合理排序，标注关键路径）
4. 创建 Sprint 计划
```

### 3.2 Sprint 计划模板

```
Sprint 目标：[一句话描述成功的样子]
时长：[2 周]
团队容量：[X 故事点]
已承诺：[Y 故事点]
缓冲：[剩余容量]

故事列表：
1. [故事] — [故事点] — [负责人] — [依赖项]

风险：
- [风险] → [缓解措施]
```

### 3.3 三种回顾格式

| 格式 | 方法 |
|------|------|
| **Start/Stop/Continue** | 开始做什么？停止做什么？继续什么？ |
| **4Ls** | Liked / Learned / Lacked / Longed For |
| **帆船模型** | 风（推进力）/ 锚（阻力）/ 礁石（风险）/ 岛屿（目标） |

## 四、干系人管理

### 4.1 权力 × 利益方格

| | 高关注度 | 低关注度 |
|--|---------|---------|
| **高权力** | **重点管理**：定期一对一，参与决策 | **保持满意**：定期更新，仅上报关键问题 |
| **低权力** | **及时告知**：定期状态更新，收集反馈 | **持续监控**：轻量级更新，按需响应 |

## 五、风险识别

### 5.1 预死亡分析（Pre-Mortem）

> 被研究证明能提升 30% 问题发现率

**三类风险**：
| 类型 | 定义 | 处理 |
|------|------|------|
| **老虎** | 真实问题，可能让项目脱轨 | 必须采取行动 |
| **纸老虎** | 被夸大的担忧 | 记录以对齐干系人 |
| **大象** | 未被说出的隐忧 | 值得深入调查 |

**紧迫程度**：
| 级别 | 含义 | 示例 |
|------|------|------|
| 发布阻断 | 发布前必须解决 | 核心功能损坏 |
| 快速跟进 | 发布后 30 天内解决 | 性能问题 |
| 持续跟踪 | 监控，必要时解决 | 边界情况 |

### 5.2 假设验证四维度

| 维度 | 问题 |
|------|------|
| **价值** | 能为客户创造价值吗？ |
| **可用性** | 用户能弄清楚如何使用吗？ |
| **商业可行性** | 市场、销售、财务能支持吗？ |
| **技术可行性** | 能用现有技术构建吗？ |

## 六、定价策略速查

| 模型 | 最适合 | 示例 |
|------|--------|------|
| 统一费率 | 简单产品 | Basecamp |
| 按席位 | 协作工具 | Slack、Figma |
| 按用量 | 基础设施、API | AWS、Twilio |
| 分级定价 | 有明显细分 | 多数 SaaS |
| 免费增值 | 有网络效应 | Spotify、Notion |
| 价值定价 | 高影响力企业 | Salesforce |

```
分层定价设计：
| 套餐 | 价格 | 目标细分 | 核心功能 |
|------|------|----------|----------|
| 免费 | ¥0 | 试用用户 | 基础功能 |
| 专业 | ¥99/月 | 个人用户 | 高级功能 |
| 企业 | ¥299/月 | 团队用户 | 全部功能 |
```

## 七、发布说明模板

```markdown
# [产品名称] — [版本 / 日期]

## 新功能
- **[功能名称]**：[1-2 句话描述及重要性]

## 改进
- **[改进方向]**：[什么变得更好了]

## 缺陷修复
- 修复了 [用户语言描述的问题]

## 破坏性变更（如有）
- **需要操作**：[用户需要做什么]
```

**撰写原则**：以用户收益为先，非技术变更；使用通俗语言；每个条目 1-3 句话。

## 八、使用规则

1. 用户要求制定 OKR、规划 Sprint、做回顾会、管理干系人时激活。
2. 数据指标体系在用户要求定义指标或 6.1 数据统计需求扩展时激活。
3. 预死亡分析在 Step 5 风险自审时作为增强工具。
4. 定价策略、发布说明在用户明确要求时使用，不影响 PRD 正文。
