# 分支流程：早期探索 + 局部修改

本文件覆盖两种非全量撰写的路径：② 早期想法/MVP 探索、③ 局部修改。

---

## Part A：早期想法与范围冻结

当用户给出早期产品想法、MVP 概念、粗略构想，尚未准备好写完整 PRD 时使用。

不要在用户已提供充分信息、要求局部修改、或说"直接写"时强制使用。

### Phase 1：探索

每轮最多问 3 个问题，只问影响产品设计的：

1. **用户**：目标用户、当前替代方案、核心任务、使用频率。
2. **商业**：免费/付费、商业模式、验证项目还是长期产品。
3. **技术**：平台、账号/登录、数据存储、第三方依赖。
4. **页面结构**：主页面/Tab、入口、导航模式、辅助页面。

**特殊用户群体探测**：若目标用户涉及老年人、儿童、残障人士、低学历/低数字素养人群、特定文化区域用户，必须在 Phase 1 追问以下适配维度（至少 1 题）：大字体/高对比度、语音输入/简化流程、无障碍合规、内容安全/家长控制。这些适配需求直接影响 5.3 的具体表现和边界场景。

用户不清楚页面结构时，提议最小页面结构草案请确认。

### Phase 2：概念版 PRD

```markdown
## 概念版 PRD

核心用户：[一句话]
要解决的一件事：[一个具体痛点]
产品形态：[平台]
页面结构：[主页面/Tab，≤5 个]
最小可用版功能（≤ 3 条）：
  1. [功能一]
  2. [功能二]
  3. [功能三]
本版本不做：[明确排除]
商业模式：[收费方式或免费]
技术前提：[账号/数据/第三方依赖]
```

然后问：`① 方向是否对齐？② 有需要调整的？确认后进入落地版。`

### Phase 3：范围冻结

```markdown
在开始落地版之前，请确认以下不会再修改：

✅ 目标用户：……
✅ 核心功能（≤ 3 条）：……
✅ 平台选择：……
✅ 页面结构：……
✅ 本版本不做：……

请回复"确认"后继续。
```

变更范围 → 返回 Phase 2。

### Phase 4：进入落地版

范围冻结后按正常起草流程（Step 2 → Step 6）写完整 PRD。页面结构整合到 5.1/5.2/5.3，不单独加章节。

### 自检

- [ ] 核心页面和导航清晰
- [ ] 每个 P0/P1 映射到至少一个页面或入口
- [ ] 用户流程含正常和异常路径
- [ ] 重要状态有进入/退出条件和可见表现
- [ ] 范围不包含已排除的功能

---

## Part B：局部修改流程（B0-B6）

当用户提供已有 PRD 并要求修改/补充/调整/删除时使用，不重走起草流程。

### 触发信号

- 用户提供了现有 PRD（文本/文件/链接）。
- 表达修改意图：改一下、补充、调整、删除、优化、增加、精简等。
- 仅局部变动，非重新起草。

大块新增功能 → 混合走：B0-B2 + 全量流程的场景树 → 大纲确认 → 起草 → 质检。

### B0 识别修改请求

读取现有 PRD。**不强套 prd-template.md 结构**，跟随原 PRD 的章节、术语、字段命名。

### B1 定位修改范围

- 位置：哪个章节、功能、段落。
- 类型：
  - 局部文本润色 → B3
  - 新增功能/子功能 → B2 + B3
  - 删除/范围收缩 → B4 + B3
  - 风险/合规补强 → B5 + B3

意图模糊时先复述定位再动手。

### B2 最小化问卷（仅新增时）

- 最多 3 题，每题带兜底选项。
- 不重走完整缺口澄清。

### B3 修改目标章节

- **其余章节原文不变**（格式、术语、编号、顺序）。
- 新增 5.3 内容使用固定三列九字段表格。
- 输出方式：默认给修改后片段；用户要求时给 diff 对比。

### B4 影响面排查

扫描联动位置：

**同功能节内联动**：修改某字段后，检查同一 5.3 表的其他字段是否需要同步更新（如改了异常场景 → 检查具体表现中的处理逻辑是否一致；改了功能入口 → 检查前置条件是否匹配）。

**跨功能节联动**：

- 四、需求列表：是否联动增删/调优先级。
- 三、需求目标：指标是否受影响。
- 5.1/5.2：流程图/原型是否需更新。
- 5.3 其他功能：是否引用了被改/删能力。
- 六、非功能需求：埋点/隐私是否联动。

**联动项不擅自改**，进末尾待办。

### B5 局部风险自审

仅对改动涉及的功能跑 `quality-review.md` Part B 的风险审查。不重审未改动部分。

### B6 输出

```markdown
## 改动说明
- 修改了什么、为什么改。

## 修改后的章节
[章节内容]

## 待办事项 & 风险问题待办

### 联动项
1. **[MED] 问题描述**
   - 建议：……
   - 下一步：……

### 风险项
1. **[LOW] 问题描述**
   - 建议：……
   - 下一步：……
```

### 硬约束

- 不重新生成完整 PRD。
- 不强套新模板，跟随原 PRD。
- 不在 PRD 主体写修改记录（除非有 1.2 变更日志则追加）。
- 不对未要求的章节"顺手优化"。
- 风险与联动只列、只建议，不擅自改。

### PRD 补充 + 原型（扩展模式）

用户要求 PRD 补充、变更表、原型图、弹窗、倒计时确认等 review-ready 材料时：

1. 读取现有 PRD/PPT/截图。
2. 提取原文档的样式、编号、术语。
3. 产出 Markdown PRD 补充（三列九字段表格）。
4. 用户要求时产出 PNG/HTML 原型。
5. 验证文案、状态、按钮、倒计时的一致性。
