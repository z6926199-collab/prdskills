# 质检：Review + 风险自审

本文件覆盖 Step 5 的全部质检工作：8 大类 Review + 5 大类风险自审。

---

## 触发时机

- PRD 全量撰写完成后自动执行（去 AI 味之后）。
- 用户主动要求"Review 这份 PRD"时执行。
- 局部修改后对改动范围执行局部质检。

## 处理规则

**覆盖度要求**：Review 必须覆盖 PRD 中的每个 5.3 功能节（5.3.1、5.3.2……），且 8 大类检查中的类型 1-6 必须对每个功能节至少扫描一遍。不允许只审 P0 漏掉 P1/P2。

| 类型 | 处理方式 |
|-|-|
| 确定性问题 | 起草模式下直接修复；Review 模式下列入清单 |
| 可改写的风险 | 信息完备 → 直接补全到 5.3 对应段落 |
| 需外部确认的 | 进「待办事项 & 风险问题待办」 |
| Review-only | 不改写原文，仅输出结论 |

严重程度：

- 🔴 **必须修复**：合规/安全/数据风险、上线阻塞、主流程断裂、无法验收。
- 🟡 **建议修复**：影响重要用户场景或评审清晰度，需确认。
- 🟢 **优化建议**：边缘场景、措辞、锦上添花。

## Part A：8 大类 Review

### 类型 1：功能完整性

| 检查项 | P0/P1 | P2 |
|-|-|-|
| 功能介绍清晰（做什么、给谁） | 必须 | 必须 |
| 入口与前置条件明确 | 必须 | 必须 |
| 用户流程覆盖正常路径 | 必须 | 必须 |
| 用户流程包含异常分支 | 必须 | 建议 |
| 页面状态列全 | 必须 | 建议 |
| 边界与降级覆盖 | 必须 | 可省略 |
| 需求列表与 5.3 一一对应 | 必须 | 必须 |

### 类型 2：边界场景覆盖

- [ ] 无网络 / 弱网 / 网络切换
- [ ] 存储空间不足
- [ ] 首次 vs 非首次使用
- [ ] 功能开关关闭时残留
- [ ] 多用户/共用设备
- [ ] 不同机型/版本/平台差异
- [ ] 多语言/区域/权限边界

### 类型 3：合规风险

- 个人数据采集明示同意（全部）
- 权限申请最小必要（全部）
- opt-in consent 设计（EU/UK）
- 儿童年龄验证（COPPA）
- 跨境数据传输（GDPR）
- 内容推荐有关闭入口（全部）
- 隐私政策需更新（新增数据采集时）

### 类型 4：升级兼容（1-to-100 / 1-to-1.5 重点）

- [ ] 老版本升级后默认行为
- [ ] 历史数据/设置保留
- [ ] 不可回滚变更标注
- [ ] 灰度策略
- [ ] 新旧版本并存兼容
- [ ] 是否影响其他模块

### 类型 5：技术依赖与逻辑一致性

5.3 描述了某能力但 6.4 未列对应依赖 → 🔴。

- [ ] 5.3 流程与 5.1 流程图一致
- [ ] 各功能节之间无状态冲突
- [ ] 优先级与 5.3 展开深度匹配
- [ ] 边界场景与前置条件不矛盾

### 类型 6：表述规范

| 检查项 | 判定 |
|-|-|
| 开发态描述（字段名/像素/状态码） | 违规 |
| AI 腔（赋能/打造/无缝/极致） | 违规 |
| 臆造内容（未确认的日期/人名/指标） | 违规 |
| 未标 `待确认` 的不确定内容 | 违规 |
| 单段超 7 句 | 建议拆分 |

### 类型 7：域专项检查

根据加载的域文件执行。未加载时跳过。

- **TV/投影仪**：焦点、遥控器、信号源、待机/OTA、CEC。
- **手机/平板**：横竖屏、分屏、折叠、权限、无障碍、全球隐私。
- **AI 语音**：唤醒→聆听→识别→理解→执行→反馈→结束全链路、话术、弱网。
- **平台配置**：查询/列表/发布/灰度/角色/回滚。

### 类型 8：证据与优先级

- [ ] P0 需求有证据或已批准假设
- [ ] 数据结论含指标定义、分母、周期
- [ ] 竞品结论引用来源或标 `待补充`
- [ ] 冲突证据未被隐藏

## Part B：5 大类风险自审

### 一、安全合规（必查）

高敏感权限（位置/相机/麦克风/通讯录/健康/生物识别）；个人数据采集（姓名/邮箱/设备ID/行为数据）；上云存储（区域/加密/保留期限）；数据跨境（SCC等合法基础）；GDPR同意/撤回/被遗忘/可携；CCPA Do Not Sell + COPPA儿童保护；第三方SDK合规边界。

### 二、多形态适配（涉及多设备时必查）

横竖屏切换、分屏最小宽度；小窗/悬浮窗支持或降级；大屏双栏/自适应；极端字号/显示设置下布局保护。域文件可扩展（TV焦点、投影仪亮度等）。

### 三、无障碍（特殊场景）

自定义控件/手势读屏器识别；动画/限时操作给足时间；仅靠颜色传达信息补语义；表单错误读屏器主动播报。

### 四、性能风险（中低端设备必查）

启动耗时（新依赖阻塞主线程）；内存占用（低内存OOM风险）；滑动流畅性（列表复杂度）；弱网（重试/超时/离线降级）；后台耗电（轮询/保活）。

### 五、其他（必查）

版本兼容：新API在最低版本的兜底。升级/降级：数据迁移、默认状态。域文件扩展区：按域文件「风险审查补充」追加。

---

## 风险等级

- **HIGH**：合规/安全/阻塞上线。
- **MED**：影响体验或部分场景，需外部确认。
- **LOW**：边缘场景，后续验收阶段处理。

---

## 输出格式

```markdown
## PRD Review 结论：[可发出 / 修改后发出 / 暂不建议发出]
🔴 必须修复：x 个 | 🟡 建议修复：x 个 | 🟢 优化建议：x 个

### 🔴 必须修复
| # | 位置 | 问题描述 | 建议修改 |
|-|-|-|-|

### 🟡 建议修复
| # | 位置 | 问题描述 | 建议修改 |
|-|-|-|-|

### 🟢 优化建议
| # | 位置 | 问题描述 | 建议修改 |
|-|-|-|-|

### 待办事项 & 风险问题待办
（按五大类分组：安全合规 / 多形态适配 / 无障碍 / 性能风险 / 其他）
每条：**[HIGH/MED/LOW] 问题描述** → 建议 → 下一步确认方

### 待确认事项汇总
| # | 章节 | 待确认内容 | 建议确认方 |
|-|-|-|-|
```

## Review 产出自评（必做）

Review 输出前，逐条检查：

1. **覆盖完整性**：每个 5.3 功能节都被类型 1-6 扫描过，无遗漏。
2. **严重程度合理性**：🔴 仅用于合规/安全/上线阻塞/主流程断裂，不滥用。
3. **每条可操作**：🔴/🟡 问题必须有「建议修改」列，不能只描述问题不给方案。
4. **待办完整**：所有需外部确认的问题都进了待办，且每条有「下一步确认方」。
5. **无臆造风险**：没有把不确定的问题标为"已解决"。

不通过 → 先修复 Review 质量再输出。

## 自查终止条件

8 类 Review + 5 类风险全部扫完。可改写的已并入 PRD；不可改写的已入待办（每条有建议与下一步）。没有风险被臆造为"已解决"。
