# 5.3 功能说明写作指南

本文件是 5.3 功能说明的唯一权威规范。起草任何 5.3 内容前必须读取本文件。

5.3 是 PRD 中研发、测试、设计直接据以执行的核心章节。写 5.3 的标准不是"我说清楚了"，而是"研发、测试、设计照此执行不会再回头来问我"。

## 一、固定三列九字段表格（不可改）

每个功能单元的 5.3 表严格使用以下 Markdown 映射。飞书原模板第三列为“原型图”；Markdown 不支持合并单元格，因此使用固定表头表达同一语义。不得新增字段行、重命名、拆分或换序。

```markdown
| 字段 | 功能描述 | 原型图 |
|-|-|-|
| 功能 | | |
| 功能详细说明 | | |
| 功能入口/前置条件 | | |
| 具体表现 | | |
| 相关定义说明 | | |
| 边界场景 | | |
| 逆向场景/退化逻辑（与前置条件相反时） | | |
| 异常场景 | | |
| 极限场景 | | |
```

不加 `交互图`、`验收标准`、`升级与迁移`、`原始依据`、`埋点`、`备注` 等额外字段行。原型图、截图或链接填第三列；没有原型时写 `待补充`。验收信息放对应功能描述中，埋点放 6.1，升级兼容放 6.2。

> 按需字段在不涉及时显式写 `不涉及（原因）`；模糊处写 `待确认`。

## 二、四层质量阶梯（L1-L4）

按功能复杂度和风险等级选择深度。先判断阶梯，再决定每个字段写到多深。

| 阶梯 | 适用范围 | 字段填写深度 |
|-|-|-|
| L1 极简 | P2 探索性功能、纯 UI 适配、文案变更 | 9 字段全列；具体表现/边界/逆向/异常/极限可写一句简述或"不涉及（原因）" |
| L2 标准 | P1 中低风险功能、迁移功能、二期补充 | 具体表现含用户流程+功能逻辑+操作反馈；异常场景至少 3 类 |
| L3 完整 | P0 核心功能、首版 MVP、有用户输入或多状态 | 具体表现完整覆盖五大子块；边界/逆向/异常/极限全部展开 |
| L4 深度 | 涉及账号/支付/隐私/跨端同步/AI/硬件协议的高风险功能 | L3 全部 + 协议/版本/容错/降级/回滚/隐私边界/安全行为 |

**升档信号**（任一命中就升一级）：

- 涉及用户输入（表单、文件、语音）→ 至少 L3
- 涉及账号/支付/订阅/会员 → 至少 L3
- 涉及多状态/状态机 → 至少 L3
- 涉及多端/跨端同步/云端协议 → 至少 L3
- 涉及 AI 推理、流式输出、模型切换 → 至少 L3
- 涉及硬件 OTA、设备绑定、固件协议 → 至少 L4
- 法律/隐私/合规边界 → 至少 L4

## 三、切分策略

5.3 不是一张大表，而是按「功能单元」重复多张三列九字段表。一张表只描述一个独立功能单元。

切分优先级（遇到第一个匹配就用）：

1. **按用户场景切**：首次使用、日常使用、错误恢复、离线场景。
2. **按页面/模块切**：列表页、详情页、设置页、弹窗。
3. **按能力切**：登录、同步、导出。
4. **按状态切**：未连接/连接中/已连接。

切分自检：

- [ ] 一个表只描述一个功能单元。
- [ ] 表与表之间的依赖在「功能入口/前置条件」里互相引用。
- [ ] 公共逻辑抽到「相关定义说明」集中定义。

## 四、字段写作精度

### 4.1 功能

动宾结构，一句话点名能力。与四、需求列表的功能点保持一致。

示例：`首次授权引导`、`弱网重试`、`语音控制结果反馈`

### 4.2 功能详细说明

一两句话：谁用、解决什么问题、范围。不写营销语言。

### 4.3 功能入口/前置条件

列出所有触发路径 + 所有前置状态。每条单独成行。

入口要写清：页面路径、触发交互（点击/长按/语音/自动触发）。

前置条件要写清：账号状态、设备状态、权限状态、数据状态、协议状态。

### 4.4 具体表现（5.3 主战场）

用户流程/状态机/字段校验/文案/关联事件在这里展开。用粗体小标题分块。**埋点不放这里，统一在 6.1 数据统计需求中。**

#### 飞书输出适配（重要）

飞书文档的表格单元格内不支持嵌套 markdown 表格。输出到飞书时：
- **用户流程 / 功能逻辑**：用编号列表写在「具体表现」单元格内，每条一行用 `<br>` 换行。
- **页面与状态**：改为编号列表形式（如 `① 关闭态：开关显示"关"，面板置灰`），每条一行用 `<br>` 换行，不使用嵌套表格。
- **字段与文案**：改为冒号分隔列表（如 `混响列表项：待确认具体名称`），每条一行用 `<br>` 换行，不使用嵌套表格。
- **关联事件**：用编号列表，每条一行用 `<br>` 换行。

**关键规则**：表格单元格内多个要点必须用 `<br>` 分隔，不要写成一整段。飞书渲染引擎会将 `<br>` 转为换行。

Markdown 输出时可保留嵌套表格。

#### 子块 1 · 用户流程（必填，含成功 + ≥2 条异常路径）

编号步骤写清每一步。区分用户行为和系统行为。失败分支单独列出。

#### 子块 2 · 功能逻辑（必填）

系统判断顺序、条件分支、保存/上报动作、幂等/冲突处理。

#### 子块 3 · 页面与状态（必填）

飞书输出时改为编号列表：`① 状态名：进入条件 → 退出条件 → 用户可见表现`。
Markdown 输出时用表格：

| 状态 | 进入条件 | 退出条件 | 用户可见表现 |
|-|-|-|-|

#### 子块 4 · 字段与文案（涉及用户输入时必填）

飞书输出时改为冒号分隔列表。
Markdown 输出时用表格：字段表（类型/必填/长度/校验/错误提示）+ 文案规则（空/加载/成功/失败/权限/网络六类场景）。

#### 子块 5 · 关联事件（按需）

上下游依赖、通知、回调、任务状态、平台配置、审核。

### 4.5 相关定义说明

集中定义术语、枚举、状态名、字段字典、文案字典。多表复用的定义只定义一次。

### 4.6 边界场景

覆盖：网络/设备/系统/账号/多端/多语言/区域/硬件能力/版本兼容。

### 4.7 逆向场景/退化逻辑

覆盖：用户取消/返回/关闭/撤销、服务降级、数据回滚、权限拒绝降级、老版本行为。

### 4.8 异常场景

覆盖：网络异常（断网/超时/4xx/5xx/401）、权限拒绝、账号异常（未登录/过期/封禁）、数据异常（空数据/格式错误/服务端返回异常）、并发冲突、存储满/权限不足、系统资源不足（内存/CPU）、后台被杀/进程异常、设备状态异常（传感器故障/电量极低）、第三方依赖失败。

### 4.9 极限场景

覆盖：重复/连续操作、大数据量/极小数据、限制上限、并发、长时间使用、跨日期。

## 五、功能形态套餐

写完用对应套餐扫一遍。所有项都落进 9 列，不新增字段。

### 套餐 A · 表单/输入类
字段类型/必填/长度/校验/错误提示；提交按钮防重复+loading态；草稿保存和恢复；极限：字符越界、特殊字符、emoji。

### 套餐 B · 列表/详情类
列表字段/排序/筛选/搜索/空态/分页；单条和批量操作；详情页来源、返回滚动保留；极限：大数据量性能。

### 套餐 C · 状态机/设备/连接类
状态枚举完整（含中间态和失败态）；中断恢复（断电/退后台/杀进程）；重试/降级链路；多设备并发。

### 套餐 D · AI/生成类
输入边界和模型选择/切换；流式输出（开始/生成中/中断/重试/重新生成）；结果消费（复制/保存/分享）；配额/计费/频率限制；内容过滤和免责。

### 套餐 E · 同步/导出/上传类
触发方式和同步范围；冲突解决策略；失败重试/断点续传；数据完整性（校验/回滚/幂等）。

### 套餐 F · 隐私授权/首次链路
协议版本和升级重新同意；拒绝路径（阻断 vs 降级）；系统权限申请时机和拒绝引导；重置/重装后重新授权。

### 套餐 G · 后台/CMS/平台配置类
列表页（排序/筛选/批量/导出）；增删改（表单校验/确认/撤销/操作日志）；状态管理（上线/下线/草稿/生效时间）；权限/角色（菜单可见性/按钮可点击）；端云联动（配置变更后客户端生效方式）。

## 六、反例与正例

1. **抽象营销话术** ❌「为用户提供更便捷的管理体验」 → ✅「用户点击导出后逐条复制到本地目录并标记已导出，失败项保留重试入口。」
2. **路径不完整** ❌「用户进入页面 → 点击按钮 → 完成」 → ✅ 完整流程含校验、权限、网络、确认、并发、失败重试、埋点。
3. **状态不闭环** ❌ 只写"处理中"和"已完成" → ✅ 列全所有状态，每状态有进入退出条件。
4. **异常合并** ❌「网络异常时提示重试」 → ✅ 区分超时、4xx、5xx、401，每类有具体文案和后续动作。

## 七、完成自检清单

- [ ] 三列表头和 9 字段全列，按需字段写 `不涉及（原因）` 或 `待确认`。字段名/顺序一致。
- [ ] 第三列始终保留；没有原型图时写 `待补充`，不删除该列。
- [ ] 至少 1 条正常路径 + 2 条异常路径。用户取消、网络异常、权限拒绝至少各一处显式处理。
- [ ] 中间态（loading/生成中/上传中）和失败态有定义且有退出条件。状态命名与文案/UI 标签一致。
- [ ] 用户输入字段有类型/必填/长度/校验/错误提示。空/加载/成功/失败/权限/网络六类场景有文案或 `待确认`。
- [ ] 数据存储位置、同步策略、持久化范围写清。埋点统一在 6.1 数据统计需求中。
- [ ] P0/P1 的边界/逆向/异常/极限四列不为空，每列至少覆盖 3 种场景。
- [ ] 术语、状态名、按钮名、文案与需求列表和 6.x 一致。与 5.1 流程图判断节点对应。
- [ ] 未发明无源数据（URL、UI 资源、最终文案写 `待确认`）。

## 八、与其他章节的引用关系

- 5.3 的「功能」字段 ↔ 四、需求列表的功能点，一一对应。
- 5.3 的页面/状态 ↔ 5.1 流程图、5.2 原型图。
- 5.3 具体表现里的埋点 ↔ 6.1 数据统计需求（埋点统一在 6.1，不在 5.3 中）。
- 5.3 边界里的隐私/权限 ↔ 6.2 隐私合规。
- 5.3 的术语/枚举 ↔ 相关定义说明字段。
- 5.3 不越位写非功能需求（性能指标、运维、上线计划在六、七章）。
- 1-to-100 / 1-to-1.5 项目：升级与迁移、老版本降级策略写入 6.2，不得增加 5.3 字段行。

## 九、轻量节奏

1. 先列出要切几张 5.3 表，每张写一行「功能」字段和优先级。
2. 选一张 P0 表，先填功能/功能详细说明/功能入口前置条件。
3. 写具体表现 - 用户流程：1 条正常 + ≥2 条异常。
4. 围绕流程补功能逻辑/页面与状态/字段与文案/关联事件。
5. 扫边界/逆向/异常/极限，按套餐 checklist 补漏。
6. 用第七节自检清单过一遍。
7. 重复处理下一张表。P1 适当压缩，P2 可只填 L1。
