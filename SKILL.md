# PRD Skills v3.4-KB

> **触发方式**：用户输入 `/tcl-cpghb-prd` 时激活本 skill。对话中提到 PRD 相关话题不会自动触发。
> **版本说明**：v3.4-KB = v3.4（PM-Master 前置分析层） + v3.3-KB（产品规划部知识库集成）。原始 v3.4 仍保留在 `/tcl-cpghb-prd`。

## 定位

以高级产品经理和需求分析师身份工作。将零散想法、功能列表、研究笔记、数据报告、竞品资料、评审意见、技术约束转化为产品、设计、研发、测试、数据、隐私和评审人可直接使用的 PRD 材料。

默认输出中文 Markdown。用户要求 DOCX、原型图 PNG 或 HTML 时再使用对应格式。

## 可选前置分析层（新增 · v3.4）

以下方法论作为 PRD 起草前的可选分析工具。当用户输入含对应信号时自动激活；不影响下文 Step 0-7 的 PRD 输出结构和硬约束。

### A. 需求分析与挖掘

当用户提供零散想法、用户反馈或只有模糊方向时激活。详见 `references/user-research.md`。

**核心工具**：
- **5W1H**：Who/Where/When/Why/What/How 六维拆解 → 映射到 PRD 2.1/2.3/5.3 等章节
- **Y 模型**：用户方案(What) → 真实动机(Why) → 重构更优方案(How)
- **JTBD 三层需求**：功能性/社交性/情感性 → 识别高价值功能点
- **伪需求识别**：追问"为什么"、观察行为、验证普遍性 → 伪需求入 Out of Scope
- **需求分析摘要模板**：核心业务目标 + 用户角色矩阵 + 功能域拆解 + KANO 分类 + 待澄清风险

### B. 产品策略分析

当用户讨论产品方向、市场定位、竞争格局时激活。详见 `references/strategy-toolkit.md`。

**核心工具**：产品策略画布、SWOT、波特五力、PESTLE、安索夫矩阵、商业模式画布、精益画布。分析结论写入 PRD 2.1 背景说明和 2.4 竞品分析。

### C. 用户研究方法

当用户提供访谈记录、NPS 数据、用户工单或要求做用户调研时激活。详见 `references/user-research.md`。

**核心工具**：JTBD 访谈脚本（开场→破冰→核心探索→追问）、访谈记录模板、老妈测试原则。

### D. 优先级评估

当用户提供功能列表但未明确优先级时激活。详见 `references/prioritization.md`。

**框架选择**：早期/PMF → ICE，有数据 → RICE，功能分类 → KANO，敏捷 → MoSCoW。分类结果映射到 PRD 四、需求列表的 P0/P1/P2。

### E. 敏捷执行与数据指标

当用户要求制定 OKR、规划 Sprint、定义指标或做风险管理时激活。详见 `references/agile-execution.md`。

**核心工具**：OKR 模板、北极星指标/AARRR、Sprint 规划、预死亡分析、干系人权力×利益方格、发布说明模板。

## 模板结构硬约束

`references/prd-template.md` 是完整 PRD 结构的唯一权威。默认执行“严格模板模式”：

1. 标题、章节、字段名、顺序和表格列数必须与模板一致。
2. 不得默认增加 `5.2.1 设计需求摘要`、`证据追溯表`、`评审更新记录`、`假设说明`等模板外章节。
3. 补充信息优先写入现有字段；确实无处承载时，放在 PRD 正文之后的“模板外补充材料”，并明确标记为不属于 PRD 模板。
4. 用户明确要求调整模板时才可增删结构，并在输出前说明偏离项。
5. 输出完成后执行模板结构校验；校验不通过不得交付。

## 交互规范

所有需要用户做选择的环节（PRD 深度、问卷澄清、产品线确认、功能大纲确认等），**优先使用可点击选项卡片呈现**，让用户直接点选而非打字。

规则：

- 每个选项包含：简短标签 + 一句话描述。
- 必须包含一个兜底选项（如"其他/都不是"），允许用户自由输入。
- 需要用户做多项选择时，标注"可多选"。
- 选项数量控制在 3-6 个，避免认知过载。
- 有推荐项时，将推荐项放在第一位并标注"推荐"。

## Step 0：请求分流 + 理解确认

> **前置可选**：若用户输入含零散想法、市场分析、用户反馈、优先级诉求等信号，先按「可选前置分析层」激活对应方法论（A~E），产出结构化分析后再进入 Step 0 路径判断。分析产物（需求摘要/策略结论/用户洞察/优先级分类）直接作为 Step 0.5 理解的输入。

### 路径判断

| 路径 | 触发条件 | 前置检查 | 执行流程 |
|-|-|-|-|
| ① 全量撰写 | 用户提供需求背景 + 功能列表 | 无 | Step 0.5 → Step 1 → Step 6 |
| ② 早期想法/MVP | 用户只有初步想法，没有明确范围 | 无 | 激活 `references/alt-flows.md` Part A，完成后进入全量撰写 |
| ③ 局部修改 | 用户提供已有 PRD + 修改要求 | **必须先收到 PRD 原文**（文本/文件/链接），未收到则索要 | 激活 `references/alt-flows.md` Part B（B0-B6） |
| ④ PRD 评审 | 用户要求 Review/评审/检查 | **必须先收到 PRD 原文**（文本/文件/链接），未收到则索要 | 运行 `references/quality-review.md`，仅输出结论 |

### Step 0.5 理解确认（所有路径必做）

**无论用户输入多少信息，都必须先执行理解确认，再进入后续流程。**

执行步骤：

1. **提炼理解**：用自己的话复述用户输入中的关键信息（背景、功能、用户、约束），让用户确认理解是否正确。
2. **识别功能歧义**：逐个功能点扫描，找出描述中的歧义、矛盾、缺失和模糊地带。重点关注：

| 歧义类型 | 示例 |
|-|-|
| 行为未定义 | "自动显示"→ 用什么传感器？什么阈值触发？走路算不算？ |
| 状态关系不清 | QS 按钮和设置页开关是否同步？不同步时谁优先？ |
| 边界未覆盖 | 通话中/全屏游戏/分屏/息屏时圆点怎么处理？ |
| 规则有矛盾 | 同时加速+转弯时圆点怎么动？ |
| 降级策略缺失 | 传感器异常/低电量/超级省电时如何降级？ |
| 术语不一致 | 同一个功能在不同位置叫不同名字？ |

3. **针对性追问**：仅问功能层面的歧义项，按影响 5.3 产出的重要性排序（最多 6 题），每题附选项卡片 + 兜底项。**不问元信息**（优先级/输出格式/PRD深度等由 skill 自动判断或在后续步骤处理）。
4. **输出「需求理解确认」**：

```markdown
## 需求理解确认

我按以下理解准备起草：

**已确认信息**：
1. 产品背景：……
2. 核心功能：……（N 个功能点）
3. 目标用户：……
4. 技术约束：……

**功能歧义项**（需确认后才能写 5.3）：
1. ……
2. ……

确认后进入功能大纲。
```

用户确认后继续。用户说「直接写」→ 将歧义项标为假设（置信度：中/低），统一写入 2.1 背景说明末尾的“待确认假设”，不得新增“假设说明”章节。

**"直接写"快捷通道**：用户说"直接写""不用问了""你自己判断"时，跳过追问环节，但必须输出「需求理解确认」并将功能歧义项标为假设（置信度：中/低），写入 2.1 背景说明末尾。纯一句话输入不允许跳过理解确认。

### Step 0.6 知识库检索选择（AskUserQuestion）

「需求理解确认」输出后、进入 Step 1 前，用 **单次** `AskUserQuestion` 调用展示检索场景。卡片左上角自动分页（1/2 → 2/2），用户一次性勾选完两页，点确定后结果全部回传。

#### 调用结构（一次调用，最多 2 题）

```
AskUserQuestion({
  questions: [
    { 页 1: 起草前检索（条件触发，≤2 选项） },
    { 页 2: 起草中检索（始终触发，3 选项） }
  ]
})
```

用户如有自定义检索需求，用 AskUserQuestion 原生输入框（选项列表下方）填入关键词，Agent 按语义搜索执行。

若 S1 和 S2 都不满足条件 → 页 1 不加入 questions 数组，卡片退化为单页（1/1）。

#### 页 1：起草前检索（条件触发）

- **出现条件**：S1 或 S2 至少一个满足触发条件；都不满足则跳过此页
- **header**：`起草前`
- **question**：`起草 PRD 前，需要检索哪些内容？（可多选）`
- **multiSelect**：true
- **选项**（按条件动态组装，label 格式 `名称: 关键词提示`）：
  - S1 满足 → `{label: "历史版本回溯: 往期SPEC/已知问题等", description: "查 {项目名} 一二期 PRD 的变更记录和已知问题"}`
  - S2 满足 → `{label: "功能规范参考: 无障碍/安全合规/卖场等", description: "查 {领域} 相关规范模板（关键词根据实际项目动态生成，如 无障碍/安全合规/卖场/多语言/深色模式 等）"}`

#### 页 2：起草中检索（始终触发）

- **header**：`起草中`
- **question**：`起草 PRD 过程中，需要检索哪些内容？（可多选）`
- **multiSelect**：true
- **选项**（始终 4 项，label 格式 `名称: 关键词提示`）：
  - `{label: "竞品与背景: 竞品分析/市场证据/项目背景等", description: "查竞品分析和背景证据，起草 2.1/2.2 时用"}`
  - `{label: "精确文档检索: 根据正文中文档名精确查询", description: "起草中遇「待确认」且涉及已命名文档时，按文件名精确检索对应文档内容"}`
  - `{label: "数据埋点与词条: FAQ/词条/埋点", description: "FAQ / 词条 / 埋点模板"}`
  - `{label: "跳过（不进行检索）", description: "跳过起草中检索"}`

#### 场景触发条件

| 编号 | 场景 | 出现条件 |
|------|------|----------|
| S1 | 历史版本回溯 | 项目为 1-to-100 或 1-to-1.5 |
| S2 | 功能规范参考 | 功能涉及 无障碍/隐私/OTA/多语言/安全/卖场/FAQ/AI |
| S3 | 竞品与背景 | 始终出现 |
| S4 | 精确文档检索 | 始终出现 |
| S5 | 数据埋点与词条 | 始终出现 |

#### 选择后执行逻辑

两题结果合并解析：
- 用户选了 **S1/S2** → Step 1 立即执行检索并注入上下文。
- 用户选了 **S3/S4/S5** → 注册到 Step 4 待执行列表，起草对应章节前执行。
- 用户通过原生输入框填写了自定义关键词 → 按语义搜索执行，结果注入对应章节。
- 两题均未选任何选项且无自定义输入 → 跳过 KB 检索，直接进入 Step 1（S0 仍可用）。

## Step 1：上下文注入

### KB 检索结果注入

若 Step 0.6 中用户勾选了 S1 或 S2，此处先静默读取已执行的 KB 检索结果，将关键信息注入当前上下文：
- S1 结果 → 注入到变更日志参考、技术约束、已知问题列表。
- S2 结果 → 注入到对应的功能参考池（无障碍规范/隐私checklist/OTA策略 等），Step 4 写 5.3 时按需取用。

### 产品策略分析（可选）

若用户在 Step 0 前已触发产品策略分析（可选前置分析层 B），或 Step 1 中用户输入涉及产品方向/竞争讨论，按需加载 `references/strategy-toolkit.md`。策略结论写入 PRD 2.1/2.4。不阻塞主流程。

### 产品线检测与域文件加载

**加载时机**：每次会话首次进入 Step 1 时执行。

**域文件分两层**：`domains/examples/`（行业领域层，单选）+ `domains/more/`（用户专属层，叠加）。两层同时生效，领域层管"这个行业怎么写"，用户层管"我们团队怎么写"。冲突时：技术知识/协议 → 领域层优先；写作风格/竞品覆盖 → 用户层优先；合规/风险审查 → 取并集。

**匹配流程**：扫描用户输入 → 匹配领域层（下表单选）→ 自动叠加用户层（如有）→ 都没有则走通用模式。匹配多个领域层时处理规则：

1. 问用户哪个是**主产品线**。
2. 主产品线域文件**完整加载**。
3. 非主产品线域文件**仅加载技术约束和合规检查**（不加载竞品列表和写作风格），避免上下文冲突。
4. 两者的技术约束/合规有冲突时，取更严格的那个。

| 关键词 | 产品线 | 域文件 |
|-|-|-|
| 手机、平板、tablet、AOSP、GMS | 手机/平板 | `domains/examples/mobile-tablet.md` |
| 电视、TV、投影仪、projector、遥控器 | TV/投影仪 | `domains/examples/tv-projector.md` |
| HDMI、CEC、EDID、信号源 | HDMI 硬件 | `domains/examples/hdmi-hardware.md` |
| 语音助手、唤醒、ASR、TTS | AI 语音 | `domains/examples/ai-voice.md` |

`domains/more/` 下的域文件在顶部用 YAML frontmatter 声明触发词（`triggers: [关键词1, 关键词2]`），匹配规则同上。

域文件生成：用户上传历史文档到 `domains/more/` 并要求"生成域文件"时，按 `references/domain-template.md` 结构提取信息并保存。后续可追加材料要求"更新域文件"。

### 项目类型判断

| 类型 | 定义 | 判断依据 | 影响 |
|-|-|-|-|
| 0-to-1（全新产品） | 全新产品，无线上版本 | 无已有产品/App | 2.2 可简写；6.2 写“不涉及（全新产品）” |
| 1-to-100（迭代优化） | 已上线功能的流程/体验优化 | 改的是已有功能的行为 | 2.2 必填；升级兼容统一写入 6.2；Review 增加升级兼容检查 |
| 1-to-1.5（新能力扩展） | 现有产品新增独立能力 | 产品已有，但新增的能力是全新的（如遥控 App 新增语音控制） | 2.2 按需；升级兼容统一写入 6.2；L 级别按新能力自身风险判断 |

**判断口诀**：产品是新的 → 0-to-1；功能是新的但产品已有 → 1-to-1.5；改的是已有功能 → 1-to-100。

**通用模式 5.3 增强**：未命中域文件时，5.3 按以下默认策略写：
- 具体表现子块优先覆盖「用户流程 + 功能逻辑 + 页面与状态」三个必填子块
- 边界/逆向/异常/极限四列按「网络 / 权限 / 数据 / 并发」四个维度逐个扫描
- 若功能涉及用户输入（表单、搜索、语音），自动升至 L3 并补充「字段与文案」子块

**常见产品线默认 L 级**：以下产品类型虽无专属域文件，但有固定风险特征，直接升档：

| 产品类型 | 默认 L 级 | 关键子块 |
|-|-|-|
| IoT / 智能家居 / 穿戴 | L3 | 设备连接状态机 + OTA + 离线降级 |
| 车载 / HUD / 车机 | L4 | 驾驶安全 + 分心检测 + 硬件协议 |
| 支付 / 订阅 / 会员 | L3+ | 扣费逻辑 + 退款 + 账号状态 |

## 5.3 硬规则

5.3 是 PRD 的核心章节，写好 5.3 的标准是"研发、测试、设计照此执行不会再回头来问"。

**唯一权威规范**：`references/feature-spec.md`。起草 5.3 前必须读取。

核心要点（完整版见 `feature-spec.md`）：

1. **固定三列九字段表格**：三列固定为“字段 / 功能描述 / 原型图”；九个字段固定为功能 / 功能详细说明 / 功能入口/前置条件 / 具体表现 / 相关定义说明 / 边界场景 / 逆向场景/退化逻辑（与前置条件相反时） / 异常场景 / 极限场景。不增行、不删行、不改名、不换序。
2. **L1-L4 深度分层**：按功能复杂度和风险等级选深度。升档信号：涉及用户输入/账号支付/多状态/多端同步/AI/硬件协议/隐私合规 → 至少 L3；涉及 OTA/固件/法律合规 → 至少 L4。
3. **切分策略**：一张表只描述一个独立功能单元。按用户场景 > 页面/模块 > 能力 > 状态 的优先级切分。
4. **具体表现五子块**：用户流程（必填，含成功 + ≥2 条异常路径）→ 功能逻辑 → 页面与状态 → 字段与文案（涉及用户输入时必填）→ 关联事件（按需）。用粗体小标题分块。**埋点不放这里，统一在 6.1 数据统计需求中。**
5. **功能形态套餐自检**：写完后按 feature-spec.md 第五节的 7 套餐（表单类/列表类/状态机类/AI生成类/同步导出类/隐私授权类/CMS配置类）逐项扫描。
6. **不可发明**：不臆造日期、负责人、机型名、URL、指标 → 统一写 `待确认`。不涉及的字段写 `不涉及（原因）`。
7. **升级兼容归位**：1-to-100 / 1-to-1.5 的升级、迁移、老版本兼容统一写入 6.2，不改变 5.3 九字段结构。

### 长输入处理

当用户输入超过 3000 字（如粘贴完整旧 PRD、多份功能文档）时：

1. **先提取结构摘要**：识别章节标题、功能列表、关键决策点，形成 500 字以内的结构化摘要。
